## 摘要
当初始用户请求未唯一指定目标实例时，自然语言实例导航变得具有挑战性。一个实用的智能体应通过主动询问仅区分目标与相似干扰物所需的信息来减轻用户负担，而非要求预先提供详细描述。现有方法往往未能实现这一目标：它们可能在充分探索替代方案之前就停在了第一个看似合理的候选上，或者即使在收集多个候选后，询问的是从单个候选得出的目标属性，而非选择用于区分候选池中实例的问题。结果，尽管进行了对话，智能体仍可能无法将目标从干扰物中区分出来，导致过早决策和冗长的用户回应。我们提出基于比较判断的主动实例导航（ProCompNav），这是一个两阶段框架，首先构建候选池，然后通过比较判断识别目标。在每一轮中，ProCompNav提取一个分割当前池的属性-值对，提出一个二元的是/否问题，并立即剪枝所有不一致的候选。这将歧义消除从开放式的目标描述重新定义为池级别的区分性提问，其中每个问题都旨在缩小候选集。在CoIN-Bench上，ProCompNav相比于具有相同最小输入的交互基线和具有详细描述的非交互基线，提高了成功率，同时大幅减少了响应长度。ProCompNav还在TextNav上达到了最先进的成功率，表明比较判断对于在相似干扰物中进行实例级导航具有广泛实用性。

## Abstract
Natural-language instance navigation becomes challenging when the initial user request does not uniquely specify the target instance. A practical agent should reduce the user's burden by actively asking only the information needed to distinguish the target from similar distractors, rather than requiring a detailed description upfront. Existing approaches often fall short of this goal: they may stop at the first plausible candidate before sufficiently exploring alternatives, or, even after collecting multiple candidates, ask about the target's attributes derived from individual candidates rather than questions selected to distinguish candidates in the pool. As a result, despite the dialogue, the agent may still fail to distinguish the target from distractors, leading to premature decisions and lengthy user responses. We propose Proactive Instance Navigation with Comparative Judgment (ProCompNav), a two-stage framework that first constructs a candidate pool and then identifies the target through comparative judgment. At each round, ProCompNav extracts an attribute-value pair that splits the current pool, asks a binary yes/no question, and prunes all inconsistent candidates at once. This reframes disambiguation from open-ended target description to pool-level discriminative questioning, where each question is chosen to narrow the candidate set. On CoIN-Bench, ProCompNav improves Success Rate over interactive baselines with the same minimal input and non-interactive baselines with detailed descriptions, while substantially reducing Response Length. ProCompNav also achieves state-of-the-art Success Rate on TextNav, suggesting that comparative judgment is broadly useful for instance-level navigation among similar distractors.

#### 1. 核心问题与整体含义（研究动机和背景）

- **问题**：在语言驱动的实例导航任务中，用户初始查询往往模糊（如“找柜子”），没有唯一指定目标实例。智能体需要通过与用户交互来消除歧义，但现有方法（如独立匹配策略）常过早停止于第一个看似正确的候选，或者虽收集多个候选，但询问的仍是基于单个候选的属性，而非能区分候选池内实例的判别性问题。这导致提前做出错误决策，或需要用户提供冗长的描述性回答，交互负担重。
- **背景**：已有工作如AIUTA采用独立匹配，对每个候选独立评分，容易受共享属性的干扰物误导。本文提出将消歧重构为**池级比较判断**：先构建完整候选池，再通过每轮选择能分割池的区分性属性值对，提出二元是否问题，一次剪枝所有不一致候选，从而避免过早决策并减轻用户负担。

#### 2. 方法论：核心思想、关键技术细节及流程

- **整体框架**：ProCompNav 是一个两阶段零样本框架。
  1. **候选池构建阶段（Pool Construction Stage）**：
     - 智能体在环境中探索，收集所有同一类别（如“柜子”）的实例。
     - 对每个检测到的实例，维护3D点云和多视角RGB视图集。通过3D点云重叠判断是否为同一候选。
     - 对每个候选的多视图进行聚类（K=6），选取代表性视图拼接为多视图拼图，并利用MLLM生成统一的**多视图描述**，总结属性-值对（如颜色、附近物体等）。
     - 当候选池大小达到阈值 \( N_{\text{min}}=5 \) 时，进入第二阶段。
  2. **递归比较阶段（Recursive Comparative Judgment, RCJ）**：
     - 初始化活跃候选集 \( U_0 = C \)。
     - 每轮迭代 \( t \)：
       - **相似性核集选择**：使用贪心剥离算法从 \( U_t \) 中选出**核集** \( G_c \subset U_t \)（内部语义相似度最大），剩余为补集 \( G_r \)。
       - **区分性属性（DA）发现**：通过LLM从核集描述的共性和NLI（自然语言推理）验证器，找出最大化 \( G_c \) 与 \( G_r \) 之间得分差异的属性值对 \( a_t^* \)。
       - **属性引导组精炼**：若 \( G_r \) 中某些候选也满足该属性（基于NLI得分阈值 \( \tau=0.9 \)），将其移入 \( G_c \)。
       - **交互式剪枝**：向用户询问“目标是否具有属性 \( a_t^* \)？”（二元是/否问题）。根据回答将 \( U_{t+1} \) 设为 \( G_c \)（回答“是”）或 \( G_r \)（回答“否”）。
       - 若 \( |U_{t+1}|=1 \) 则判定为目标；否则继续剪枝。若回答“否”但 \( G_r \) 为空，则重新探索增加候选。
- **关键技术细节**：
  - **相似度计算**：平均文本和视觉嵌入的余弦相似度（DINOv2视觉嵌入 + 描述文本嵌入）。
  - **NLI验证器**：使用DeBERTa-v3-large，将描述和假设（“实例具有属性a”）输入，通过logits计算蕴含得分，提供稳定的组级对比。
  - **池化匹配预训练**：针对非交互场景（TextNav），从文本目标直接提取属性集合，每轮固定取核集，无需用户反馈。

#### 7. 优点

- **创新点**：将实例导航中的消歧从独立匹配重新定义为**池级比较判断**，通过递归剪枝实现高效、低负担交互。
- **用户友好**：只需二元是否回答，大幅降低用户认知和语言负担（RL低至4.2 token）。
- **零样本且通用**：无需训练，可迁移至非交互（详细描述）场景，在TextNav实现SOTA。
- **技术细节完善**：采用多视图聚合提升候选描述质量；NLI验证器提供稳定的属性评分；贪心核集选取保证区分性属性发现的有效性。
- **充分消融**：通过多组控制实验验证了每个组件的必要性（如NLI优于仅LLM、相似性核集优于KMeans等）。

#### 8. 不足与局限

- **仿真局限**：所有实验在模拟器中进行，**未在真实机器人上部署**，真实环境中的传感器噪声、物体多样性等问题未考虑。
